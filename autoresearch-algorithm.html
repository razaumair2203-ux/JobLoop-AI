AutoResearch Algorithm

Karpathy loop adapted for prompt optimization | Validated April 30, 2026

1. Architecture: 3 Files (Karpathy-Faithful)

EDITABLE (AI modifies this)
cv-generation.ts
OR jd-parser.ts — ONE file per session. The AI agent proposes modifications to the prompt instructions.
READ-ONLY (Ground Truth)
test-bank/
50 CV/JD pairs + human-verified ideal outputs. Split into 20 train / 15 validation / 15 held-out. NEVER modified by the loop.
RULES (Agent Instructions)
program.md
What the AI agent can/cannot do. Mutation operators, constraints, history format, stop conditions.

2. The Main Loop

1 START: Load Current State
Read results.tsv (last 20 + top 5 best) + current prompt file
2 SELECT Mutation Operator
Rotate through 6 operators. Log which one is used. If plateau detected, force RESTRUCTURE.
3 AI Agent Modifies Prompt
Agent reads program.md rules + history + selected operator, applies ONE mutation to prompt file
4 Git Commit the Change
git add cv-generation.ts && git commit -m "[autoresearch] {mutation_type}: {description}"
5 Run Against Training Set
Pick 10 random pairs from 20 training pairs. Run modified prompt on each. Collect outputs.
6 Score with Composite Scorecard
Compute checks_avg (structural checks passed / total) and bert_avg (mean BERTScore F1) per pair. Average each across 10 pairs. Track separately.
7 BOTH checks_avg AND bert_avg ≥ Previous Best?
YES
KEEP
Advance branch. This prompt becomes the new baseline. Reset consecutive_discards = 0.
NO
DISCARD
git reset --hard HEAD~1. Revert to previous prompt. consecutive_discards++.
8 Log to results.tsv
[version, checks_avg, bert_avg, mutation_type, status, description]
9 Every 10th Iteration?
YES
Validate Against 15 Val Pairs
Full validation run. If val score drops while train rises = OVERFITTING. Stop and alert.
NO
Continue
Skip validation, proceed to stop check.
10 Stop Condition Met?
iteration >= 50 OR consecutive_discards >= 20 OR overfitting_detected
YES
STOP
Log final summary. Report best score, total keeps, improvement %.
NO
LOOP BACK to Step 2
Select next mutation operator and repeat.

3. Composite Scorecard (The Metric)

Why not edit distance? No prompt optimization framework uses it. It penalizes valid paraphrases ("Led 8 engineers" vs "Managed 8 engineers" = high edit distance, same quality). We use structural checks + BERTScore instead — deterministic, non-circular, semantically meaningful.
checks_avg = checks_passed / total_checks     bert_avg = mean(BERTScore_F1 per pair)

KEEP if checks_avg >= prev AND bert_avg >= prev
No invented weights. No thresholds. Just: did BOTH improve?
Structural Checks — tracked separately
BERTScore F1 — tracked separately

cv-generation.ts Scorecard

S1
All JD-required skills addressed?
Method: count(jd_skills in cv_output) / count(jd_skills)
S2
Each skill claim has Cloud evidence?
Method: set(cv_skills) subset_of set(cloud_skills) — no fabrication
S3
Metrics/numbers preserved from source?
Method: regex extract numbers from source CV, check present in output
S4
Within page limit (2 pages)?
Method: word_count(output) <= 800
S5
No fabricated skills?
Method: set(output_skills) - set(source_cv_skills + cloud_skills) == empty
S6
Action verbs on bullet starts?
Method: each bullet[0] is verb (Led, Built, Reduced, Designed...)
S7
ATS-parseable structure?
Method: has standard sections (Experience, Education, Skills), reverse chronological
S8
Company names, dates, titles preserved?
Method: exact_match(source.companies, output.companies)
B1
Bullet quality vs human ideal
Method: BERTScore F1 of generated bullets vs human-written ideal bullets

jd-parser.ts Scorecard

S1
All requirements extracted?
Method: count(extracted_reqs) / count(ideal_reqs) per JD
S2
Must-have vs nice-to-have correct?
Method: classification_match(extracted.type, ideal.type) per requirement
S3
Company name extracted correctly?
Method: exact_match(extracted.company, ideal.company)
S4
Role level correctly identified?
Method: exact_match(extracted.seniority, ideal.seniority)
S5
No phantom requirements invented?
Method: set(extracted_reqs) - set(ideal_reqs) == empty
S6
Location / remote status correct?
Method: exact_match(extracted.location, ideal.location)
S7
Experience years extracted?
Method: numeric_match(extracted.years, ideal.years, tolerance=0)
B1
Responsibilities text quality
Method: BERTScore F1 of extracted responsibilities vs human-written ideal
The Langfuse Warning: "If it's not measured, it gets cut." Their optimizer removed entire features because test cases didn't cover them. Every feature we care about MUST have a scorecard check. If it's not in the scorecard, the loop WILL remove it.

4. Six Mutation Operators

Each iteration, the agent selects one operator (rotating). Logging which operator produced keeps vs discards reveals which mutation types are most productive.

1
ADD_CONSTRAINT
Add a specific rule or boundary condition to the prompt
"Never use more than 5 bullets per role"
2
ADD_NEGATIVE
Add a "do NOT do this" example showing what to avoid
"BAD: Leveraged synergies to optimize..."
3
RESTRUCTURE
Reorganize prompt sections, reorder instructions
Move quality checklist before output format
4
TIGHTEN_LANGUAGE
Replace vague instructions with precise, specific ones
"add metrics" -> "prefix each bullet with a number"
5
REMOVE_BLOAT
Delete redundant, low-value, or over-specified instructions
Remove duplicate rules, trim verbose explanations
6
ADD_COUNTEREXAMPLE
Add an edge case example showing correct handling
"If candidate has no metrics: describe scope instead"

5. Test Bank: 50 Pairs (20/15/15 Split)

Research-validated: Decagon found 20-100 is optimal. 500 samples actively HURTS (-2% performance, 75% longer prompts). 50 is in the sweet spot.
20
TRAIN
Random 10 selected each iteration. Optimization runs against these. The loop sees this data.
15
VALIDATION
Checked every 10th iteration. Detects overfitting: if val drops while train rises = STOP.
15
HELD-OUT
NEVER touched during optimization. Final quality gate only. Run once after loop completes.

9. Validation Gate (Before Committing to Nightly)

"Prompt Optimization Is a Coin Flip" (April 2026, arXiv:2604.14585) — 49% of optimization runs showed improvement, statistically indistinguishable from random. We MUST prove the loop works before running it nightly.
Run 20 Manual Iterations
One-time test. Takes ~1 day of compute.
Compare: Optimized vs Original Prompt
Run both on held-out 15 test pairs. Collect scores.
ANOVA Test
Is the improvement statistically significant?
p < 0.05?
YES (p < 0.05)
Proceed with Nightly Runs
Loop delivers statistically significant improvement. Worth the compute.
NO (p >= 0.05)
STOP. Investigate.
Loop may not help for this prompt. Check: test bank quality? Scorecard coverage? Prompt already near ceiling?

10. Full Algorithm (Pseudocode)

function runAutoResearch(config: { targetPrompt: 'cv-generation' | 'jd-parser', maxIterations: 50, maxConsecutiveDiscards: 20, validationInterval: 10, trainBatchSize: 10 }) { // --- SETUP --- const testBank = loadTestBank() // 50 pairs total const { train, validation, heldOut } = splitTestBank(testBank, [20, 15, 15]) const history = loadResults('results.tsv') const scorecard = getScorecard(config.targetPrompt) let currentPrompt = readFile(config.targetPrompt + '.ts') let bestScore = scorePrompt(currentPrompt, train, scorecard) let consecutiveDiscards = 0 let iteration = 0 const mutationOrder = [ 'ADD_CONSTRAINT', 'ADD_NEGATIVE', 'RESTRUCTURE', 'TIGHTEN_LANGUAGE', 'REMOVE_BLOAT', 'ADD_COUNTEREXAMPLE' ] // --- MAIN LOOP --- while (iteration < config.maxIterations) { iteration++ // Step 1: Select mutation operator let mutation = mutationOrder[iteration % 6] if (consecutiveDiscards >= 10) mutation = 'RESTRUCTURE' // plateau override // Step 2: AI agent modifies prompt const contextForAgent = { currentPrompt, recentHistory: history.slice(-20), topResults: history.sortBy('score').slice(0, 5), mutation, rules: readFile('program.md') } const modifiedPrompt = await askAgent(contextForAgent) // Step 3: Git commit writeFile(config.targetPrompt + '.ts', modifiedPrompt) gitCommit(`[autoresearch] ${mutation}: ${modifiedPrompt.description}`) // Step 4: Score against random training batch const batch = randomSample(train, config.trainBatchSize) // 10 of 20 const { checks_avg, bert_avg } = scorePrompt(modifiedPrompt, batch, scorecard) // Step 5: Keep if BOTH dimensions improved (no weights, no thresholds) if (checks_avg >= best.checks_avg && bert_avg >= best.bert_avg) { best = { checks_avg, bert_avg } consecutiveDiscards = 0 log(history, { iteration, checks_avg, bert_avg, mutation, status: 'keep' }) } else { gitReset() // revert to previous prompt consecutiveDiscards++ log(history, { iteration, checks_avg, bert_avg, mutation, status: 'discard' }) // Plateau: reset to best known prompt if (consecutiveDiscards >= config.maxConsecutiveDiscards) { currentPrompt = getBestPromptFromHistory(history) consecutiveDiscards = 0 // give it another chance } } // Step 6: Periodic validation if (iteration % config.validationInterval === 0) { const val = scorePrompt(currentPrompt, validation, scorecard) if (val.checks_avg < best.checks_avg - 0.05 || val.bert_avg < best.bert_avg - 0.05) { // Validation diverging from training on either dimension = overfitting log('OVERFITTING DETECTED: train=' + JSON.stringify(best) + ' val=' + JSON.stringify(val)) break } } } // --- FINAL EVALUATION --- const final = scorePrompt(currentPrompt, heldOut, scorecard) log('FINAL: held-out checks_avg=' + final.checks_avg + ' bert_avg=' + final.bert_avg) return { best, final, totalKeeps: history.filter(h => h.status === 'keep').length } }
function scorePrompt(prompt, testPairs, scorecard) { const scores = testPairs.map(pair => { // Run the prompt against this test pair const output = await runLLM(prompt, pair.cv, pair.jd) // Structural checks (deterministic, code-only) const checks = scorecard.map(check => check.evaluate(output, pair)) const structuralScore = checks.filter(c => c.passed).length / checks.length // BERTScore (frozen BERT model, non-circular) const bertScore = computeBERTScore(output.freeText, pair.ideal.freeText) return { structuralScore, bertScore } }) return { checks_avg: average(scores.map(s => s.structuralScore)), bert_avg: average(scores.map(s => s.bertScore)) } }

11. What This Is NOT

NOT training a model. The LLM (Claude/GPT) is frozen. We're optimizing the INSTRUCTIONS we give it, not its weights.
NOT using user data. The test bank is OUR synthetic data. User CVs, JDs, and outcomes never enter the loop.
NOT LLM-as-judge. Structural checks are code. BERTScore uses a frozen BERT model. Neither uses the same LLM that generates output.
NOT guaranteed to work. 49% of optimization runs show no improvement. ANOVA pre-test required before nightly runs.

12. Implementation Checklist

[ ] Test bank — 50 CV/JD pairs with human-written ideal outputs (the hardest part)
[ ] Scorecard functions — 8 structural checks for cv-gen, 7 for jd-parser (pure TypeScript)
[ ] BERTScore integration — Python subprocess or sentence-transformers API
[ ] program.md — Rules for what the AI agent can/cannot modify
[ ] Loop orchestrator — The main loop with git, scoring, logging
[ ] results.tsv logger — Append-only, tab-separated
[ ] ANOVA validation gate — 20-iteration pre-test before committing to nightly
[ ] Cron trigger — Nightly schedule (after validation gate passes)
Priority order: Test bank first (blocks everything else) -> Scorecard functions -> Loop orchestrator -> program.md -> Validation gate -> Cron. The test bank is the bottleneck — without good human-written ideals, the loop optimizes toward the wrong target.
