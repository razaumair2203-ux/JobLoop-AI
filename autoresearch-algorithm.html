2. The Main Loop (Actual Implementation)

1 LOAD: Deterministic Eval Batch
Sort ALL training pairs by ID. Take first 10. Same batch every run -- no randomness between restarts. Load current prompt file.
2 SELECT Mutation (Effectiveness-Weighted)
First 6 iterations: round-robin (try each operator once). After that: select by keep rate + exploration bonus for under-tried types. Never repeat last type. Plateau (10+ discards): force RESTRUCTURE.
3 LLM Mutates Prompt
Call DeepSeek V4 Pro with mutation directive + current prompt + recent failures. Response stripped of markdown wrappers. max_tokens=8192.
4 Generate CVs with Mutated Prompt
Run mutated prompt against 10 training pairs. response_format: { type: "json_object" } enforced. Incumbent outputs cached (only regenerate on KEEP).
5 Score with Gated Scorecard (8 checks)
Per pair: run 8 deterministic checks, each with calibrated threshold. Per-pair gate: pair passes if >= 6/8 checks pass (75%). Aggregate gate: pass if >= 70% of pairs pass their per-pair gate.
6 compareScorecards(incumbent, challenger)
Gate 1 pass/fail -> BERTScore tiebreaker (if available) -> check count -> float score avg (0.02 deadband) -> true tie = incumbent wins
CHALLENGER WINS
KEEP
Mutated prompt becomes new incumbent. Reset consecutive_discards = 0. Save prompt version. Invalidate incumbent cache.
INCUMBENT WINS
DISCARD
Revert to incumbent prompt. consecutive_discards++. Log mutation type as failed.
7 Run Safeguard Checks
5 checks: regression detection, overfitting (train vs val divergence), prompt drift (similarity to original), exploit detection, ANOVA pre-test on held-out set
8 Stop Condition?
iteration >= 50 OR consecutive_discards >= 20 OR overfitting_detected
YES
STOP + ANOVA Pre-Test
Run Welch's t-test + Cohen's d on held-out set. Report statistical significance and effect size before deploying.
NO
LOOP BACK to Step 2
Select next mutation operator based on effectiveness weights.

3. Gated Scorecard (Calibrated from Data)

Key design principle: NO LLM judge. All 8 checks are deterministic code. All checks compare against RAW HUMAN-WRITTEN SOURCE TEXT, not LLM output. This makes the scorecard ZERO-CIRCULAR -- it cannot be gamed by the AI generating the output.

Decision Logic (4-level cascade)

1. Gate 1 pass/fail -- if only one variant passes, that one wins
2. BERTScore F1 tiebreaker (if both pass Gate 1, +0.01 deadband)
3. check_count -- more checks passed = closer to viability
4. float_score_avg -- finer-grained comparison (+0.02 deadband)
5. True tie = incumbent wins (conservative: don't change what works)

Per-Pair vs Aggregate Gates

// Per-pair gate: a pair passes if >= 75% of its checks pass const PER_PAIR_THRESHOLD = 0.75; // 6/8 checks minimum // Aggregate gate: pass if >= 70% of pairs pass their per-pair gate const AGGREGATE_THRESHOLD = 0.70; // 7/10 pairs minimum // This is NOT "all checks on all pairs must pass" // A stretch application with low JD coverage but perfect // factual preservation can still pass its per-pair gate.

Threshold Calibration (Empirical, Not Guessed)

Method: Ran all 45 ideal outputs through the scorecard. Computed per-check score distribution (min, P10, P25, median, P75, mean). Set thresholds at P10 -- anything below the 10th percentile of IDEAL outputs is genuinely bad, not a scorecard artifact. This is the same principle DSPy uses: calibrate from data, not from intuition.

4. Six Mutation Operators (Effectiveness-Weighted)

Selection strategy based on PromptBreeder (DeepMind 2024): track operator effectiveness, boost operators with higher keep rates, explore under-tried operators. NOT simple round-robin.

7. Claim Validation: Honest Assessment

Every claim about this system validated against actual code, actual data, and published research. No hand-waving.

Claim: "Thresholds are data-driven"

Are thresholds empirically calibrated?
Evidence: calibrate-thresholds.ts runs all 45 ideal outputs through scorecard, computes P10 per check. Thresholds set at P10 floor.

Counter-evidence: P10 as the percentile choice is ITSELF a practitioner judgment. Why not P5 or P15? And 45 pairs is a small sample -- P10 of 45 values = 4.5th value, which is noisy.

DSPy comparison: DSPy doesn't use fixed thresholds at all -- it uses float scores for optimization and bool for bootstrapping. Our P10 approach is a reasonable middle ground but NOT "optimal."
PARTIALLY VALIDATED -- Better than guessing, but P10 choice is still judgment. Will improve with more data.

Claim: "Scorecard is hallucination-proof"

Can the LLM game the scorecard?
Evidence: All 8 checks compare against raw_cv_text (human-written), not against LLM-generated expected_output. The LLM generating the CV never sees the scorecard code.

Counter-evidence: The LLM mutating the PROMPT can indirectly optimize for scorecard artifacts. E.g., ADD_CONSTRAINT "always repeat the exact company name" would boost C8 (factual_preservation) without improving CV quality. The scorecard can't detect this -- it's a valid pass.

Mitigation: The mutation agent doesn't see scorecard implementation details -- only failure names. Safeguards check for prompt drift. But this is defense-in-depth, not proof.
PARTIALLY VALIDATED -- Zero-circular scoring is real. But prompt mutations can still game check heuristics indirectly.

Claim: "Gates should be hardcoded, not intelligent"

Why not use an LLM to judge quality?
For hardcoded:
- Deterministic: same input = same output, always
- Zero cost: no API calls for scoring
- No circularity: code can't hallucinate
- Reproducible: threshold calibration is auditable

For intelligent (LLM judge):
- Could catch nuance (tone, flow, persuasiveness)
- More aligned with actual human judgment

Research: DSPy (Stanford): uses programmatic metrics. MIPRO: uses LLM as meta-optimizer but programmatic scorer. ExtractBench: per-field typed checks, no LLM judge. PromptBreeder: binary self-play (LLM evaluates LLM -- circular). DeepEval (Datadog): LLM judge for claim-level verification -- but requires ground truth claims list.

Industry consensus: For optimization loops, programmatic metrics are preferred. LLM judges introduce variance and cost. LLM judges are used for FINAL evaluation (human proxy), not in-loop optimization.
VALIDATED -- Hardcoded gates are the correct choice for optimization loops. LLM judges belong in final evaluation, not in the keep/discard decision.

Claim: "AutoResearch will improve prompt quality"

Does the loop actually work?
Evidence FOR:
- Karpathy's original loop: proven on code generation tasks
- DSPy: 20-100 examples + optimization = measurable improvement on BBH, MATH, etc.
- PromptBreeder: evolving prompts outperforms static prompts on GSM8K, SVAMP
- Our scorecard: 97.8% of ideal outputs pass gate (baseline is sound)

Evidence AGAINST:
- total_iterations = 0. We have NEVER RUN the loop. All claims are theoretical.
- Prompt-model mismatch risk: optimizing on DeepSeek Pro, deploying on Flash
- 45 pairs may be insufficient for robust optimization (DSPy recommends 30-300)
- Kaggle data quality: metrics_preserved fails 44% at 30% threshold due to data, not prompt

What we'll know after first run:
- Actual keep rate (expected 10-20%)
- Which mutation operators produce keeps
- Whether val tracks train (overfitting detection)
UNPROVEN -- Architecture is sound by research standards. But total_iterations=0. First run is the real test.

Claim: "requirementAddressedInText is now correct"

Does the multi-word fix actually work?
Before fix: ANY single domain word match = "addressed". "machine shop" matches "machine learning" requirement.

After fix: Requirements with 3+ domain words require 2+ matching words. Short requirements (1-2 words) still use single-word match.

Result: jd_requirements fail rate 7% -> 20% on ideal outputs. Stricter but more accurate. Per-pair gate still 97.8%.

Limitation: Still uses simple string inclusion, not semantic matching. "React Native" and "ReactJS" would not match. Synonym handling would require a thesaurus or embedding lookup -- deferred.
VALIDATED -- Multi-word matching is correct for the current scope. Semantic matching is a future improvement.

Claim: "Effectiveness-weighted selection beats round-robin"

Is the new mutation selection better?
Research basis: PromptBreeder (DeepMind 2024) evolves mutation operators themselves, tracking which operators produce improvements. Karpathy's original uses agent-guided selection.

Our implementation: Track keep rate per operator type. After baseline phase (6 iterations), select highest keep rate + exploration bonus for under-tried types. Never repeat last type.

Limitation: With total_iterations=0, we have no keep rate data. The algorithm degenerates to round-robin until we have data. Cannot validate effectiveness until first run completes.

Theoretical advantage: If ADD_CONSTRAINT produces 3/3 keeps and REMOVE_BLOAT 0/3, round-robin wastes iterations on REMOVE_BLOAT. Weighted selection doubles down on what works.
THEORETICALLY SOUND -- Cannot validate empirically until loop runs. Research supports the approach.
