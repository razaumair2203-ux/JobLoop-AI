3b. CV Scorecard: 8 Checks (All Zero-Circular)

Key design: ALL 8 checks compare against raw_cv_text (human-written source) or structural properties of the output. NONE compare against LLM-generated expected_output. This makes the scorecard zero-circular.

Hard gates vs soft scores (May 10 fix): C2 (No Fabrication), C5 (No Fabricated Skills), C8 (Factual Preservation) are hard gates — any failure = pair fails instantly. Remaining checks (C1, C3, C4, C6, C7) are soft scores — averaged into a float, pair passes if average ≥ 0.50. When raw_cv_text is insufficient (<50 chars), hard gate checks return "skipped" (pass with score=1) instead of falling back to circular AI-vs-AI comparison.

Checks 1-4

C1
JD Requirements Coverage
Two-layer: (A) each JD requirement topic addressed? Domain nouns extracted, filler/stopwords skipped. 3+ word requirements need 2+ word matches. (B) Technical terms from raw JD (50+ tech regex). Only ADDRESSABLE requirements count -- stretch apps not penalized.
Threshold: ≥ 55% | Score: reqScore*0.7 + techScore*0.3 | Circularity: NONE
C2
No Fabrication
Every skill in generated skills{} must exist in raw_cv_text. Token-level fallback: "machine learning" splits to ["machine","learning"], any match passes.
Threshold: ≥ 85% skills verified | Circularity: NONE
C3
Metrics Preserved
Numbers from raw_cv_text (excluding dates/years) must appear in generated output. Extracts: "$2M", "35%", "500K", count nouns ("9 aircraft").
Threshold: ≥ 30% | LOW because Kaggle pairs don't share numbers | Circularity: NONE
C4
Word Count
Total words in generated CV ≤ 800 (2-page equivalent).
Threshold: binary pass/fail | Circularity: NONE

Checks 5-8

C5
No Fabricated Skills
Technologies in experience bullets (60+ tech regex: Python, AWS, Docker, etc.) must exist in raw_cv_text.
Threshold: 0 fabricated | Circularity: NONE
C6
Action Verbs
Bullets should start with action verbs. 200+ verb vocabulary (past/present/gerund). Handles "CompanyName: Built..." label patterns and "Co-authored" hyphenation.
Threshold: ≥ 50% bullets | Circularity: NONE
C7
ATS Structure
Must have: summary, experience (with company+title+dates+bullets per role), skills section.
Threshold: 0 issues (binary) | Circularity: NONE
C8
Factual Preservation
Company names (normalized, core-name extraction) and dates (year-level matching) from generated CV must exist in raw_cv_text.
Threshold: ≥ 85% | Circularity: NONE
B1
BERTScore F1 (Placeholder)
Semantic similarity. Requires Python subprocess. NOT YET IMPLEMENTED.

4b. JD Scorecard: 9 Checks (Calibrated from N=38 Trial)

CALIBRATION SOURCE: Thresholds set from P10 of a trial run (N=38 JDs parsed by Llama-3.3-70B via NIM). The trial model differs from the loop model (DeepSeek V4 Flash) -- score distributions may differ. Thresholds should be re-calibrated after first DeepSeek Flash run.

Checks 1-5 (Scalar Fields)

J1
Company
Multi-strategy matching: exact → contains → word overlap (Jaccard) → Levenshtein → Jaro-Winkler. Best score wins. JW added May 10 — better for names due to prefix weighting ("Thompson" vs "Thomson" = JW 0.96 vs Lev 0.88). Handles subsidiaries via word overlap. Both-empty = pass.
Threshold: ≥ 40% | Circularity: NONE
J2
Title
Same multi-strategy matching (5 strategies, best wins). Jaro-Winkler helps with prefix-similar titles. "Senior Software Engineer" vs "Sr. Software Eng." scores high via word overlap.
Threshold: ≥ 40% | Circularity: NONE
J3
Location
Same 5-strategy matching. "Denver, USA" vs "Denver, CO" scores 67% via word overlap (was 53% Lev-only). "Remote" vs "" = extra extraction, passes.
Threshold: ≥ 45% | Circularity: NONE
J4
Experience Years (Min)
Exact match = 1.0. Within ±1 = 0.8 (passes if threshold ≤ 0.8). Both null = pass. One null = fail.
Threshold: ≥ 80% | Circularity: NONE
J5
Experience Years (Max)
Same logic as J4. Many JDs don't specify max, so both-null is common and passes.
Threshold: ≥ 80% | Circularity: NONE

Checks 6-9 (List Fields)

J6
Requirements Recall
For each expected requirement: extract words >3 chars, check if any parsed requirement contains 2+ matching words (for 3+ word requirements) or 1+ word (for short requirements). Counts how many expected requirements were found.
Threshold: ≥ 40% recall | Circularity: MEDIUM (GT is AI-generated)
J7
Requirements Precision (No Phantoms)
For each parsed requirement: check if it matches any expected requirement (same word-overlap matching). Counts how many parsed requirements are legitimate (not hallucinated).
Threshold: ≥ 85% precision | Circularity: MEDIUM
J8
Requirement Types
For matched requirement pairs: does parsed type (must_have/nice_to_have) match expected type? Conservative bias: must_have when should be nice_to_have = 0.5 partial credit (being cautious is better than being lenient).
Threshold: ≥ 80% accuracy | Circularity: MEDIUM
J9
Responsibilities Recall
Same word-overlap recall as J6, but for responsibilities list. Requires min(2, word_count) matching words per expected responsibility.
Threshold: ≥ 45% recall | Circularity: MEDIUM

5. compareScorecards(): The Keep/Discard Decision

Both loops use this exact same function. It takes two ScorecardResult objects (incumbent + challenger) and returns a winner. This is the ONLY place where keep/discard decisions are made.
function compareScorecards(incumbent, challenger): // Level 1: Gate verdict (primary filter) if incumbent.gate1_verdict == "pass" && challenger.gate1_verdict == "fail": return "incumbent" if incumbent.gate1_verdict == "fail" && challenger.gate1_verdict == "pass": return "challenger" // KEEP // Level 2: BERTScore tiebreaker (both pass Gate 1) if both pass && bertscore available: if challenger.bertscore > incumbent.bertscore + 0.01: return "challenger" // 1% deadband prevents noise // Level 3: Check count (both fail or both pass, no BERTScore) if challenger.gate1_passed > incumbent.gate1_passed: return "challenger" // More checks = closer to viability // Level 4: Float score average (finer-grained) if challenger.legacy_structural_avg > incumbent + 0.02: return "challenger" // 2% deadband // Level 5: True tie = incumbent wins (conservative) return "incumbent"
Why conservative tiebreaking? In prompt optimization, changing the prompt has COST (drift risk, regression risk, complexity). A challenger must demonstrate CLEAR improvement to justify the change. Ties go to stability.

6. Aggregation: Per-Pair Results to Final Verdict

The Two-Level Gate

// LEVEL 1: Per-pair gate // Each pair's scorecard has N checks (8 for CV, 9 for JD) // A pair PASSES if >= 75% of its checks pass const PER_PAIR_THRESHOLD = 0.75; // For CV (8 checks): 75% of 8 = 6.0 -> need >= 6/8 // For JD (9 checks): 75% of 9 = 6.75 -> need >= 7/9? NO! // Actually: code checks gate1_passed / gate1_total >= 0.75 // So 6/9 = 0.667 < 0.75 = FAIL. 7/9 = 0.778 >= 0.75 = PASS. // JD pairs need 7/9 checks to pass their per-pair gate. // LEVEL 2: Aggregate gate // Count how many pairs passed their per-pair gate const AGGREGATE_THRESHOLD = 0.70; // For 10 pairs: 70% = 7.0 -> need >= 7/10 pairs passing
UPDATED May 10 — Hard Gates + Soft Scores: buildResult() now splits checks into two categories:

Hard gates (no_fabrication, no_fabricated_skills, factual_preservation): ANY failure = pair fails instantly, regardless of soft scores. These are integrity checks — a fabricated CV is unusable no matter how well-structured it is.

Soft checks (all other checks): averaged into a float score. Pair passes if hard gates all pass AND soft average >= 0.50.

The aggregator still uses the per-pair pass/fail ratio (>= 70% of pairs must pass). But now "pass" means something more meaningful: integrity guaranteed + quality adequate.

Research basis: ExtractBench, CoNLL-2003, and SemEval all use per-field scoring with field-appropriate thresholds. The 2025-2026 consensus (Evidently AI, ICLR 2025 LLM-as-Judge study) recommends hybrid: deterministic for structural/verifiable properties + separate treatment for semantic properties.

FieldHow ComputedUsed For

8. Independent Audit: What's Real vs What's Claimed

Scoring System Audit (May 10, 2026)

DATA-INFORMED
"Thresholds are research-backed" -- They are set at P10 of ideal output distributions. P10 as the percentile choice is practitioner judgment, not a recognized research standard. DSPy doesn't use fixed thresholds. ExtractBench uses Levenshtein >= 0.8 (which IS a published standard). Our P10 choice is reasonable but arbitrary.
CORRECT
"Scorecard is zero-circular" -- CV gen checks compare against raw_cv_text. JD scalar checks (J1-J5) compare parsed vs human-corrected GT. Check LOGIC cannot be gamed by the AI. However, JD list checks (J6-J9) have MEDIUM circularity because GT requirements/responsibilities were AI-generated.
PARTIALLY VALID
"JD thresholds calibrated from trial" -- Yes, but trial model (Llama-3.3-70B) differs from loop model (DeepSeek V4 Pro). Score distributions may differ. Trial used richer scoring (city-level location, subsidiary-aware company) that has now been ported to scorecard via word-overlap matching (Fix #2, May 10). N=38 is small for P10 calibration (3.8th value).
FIXED
"selectTrainingBatch is deterministic" -- WAS random (Fisher-Yates shuffle). FIXED May 10, 2026: now sorts by ID and takes first N. Previous audit claimed it was fixed but code was never changed.
FIXED
"Trial and scorecard scoring match" -- Trial used richer matching (word-overlap + city extraction) for company/location. Scorecard used basic Levenshtein. FIXED May 10, 2026: scorecard checkFuzzyField() now uses multi-strategy matching (exact -> contains -> word overlap -> Levenshtein, best score wins).
ADEQUATE
"40% company threshold is useful" -- At 40%, almost anything passes. This is by design: company extraction is informational, not a hard gate. The per-pair gate (7/9 checks) means company failure alone won't kill a pair. With word-overlap matching, 40% catches only truly garbage parsing.
UNPROVEN
"The loop will improve prompt quality" -- total_iterations = 0. Architecture follows published patterns (Karpathy, DSPy, PromptBreeder). Scorecard calibration is sound. But zero empirical evidence that our specific implementation produces improvements on our specific task.
CORRECT
"No LLM judge, no cost for scoring" -- All checks are deterministic code. Zero API calls for scoring. Zero circularity in check logic. Zero variance between runs (given same inputs).

Alternative Approaches Considered

ExtractBench (ContextualAI, Feb 2026)

Per-field typed scoring with fixed Levenshtein >= 0.8
Difference: ExtractBench uses a FIXED threshold (0.8) for all fuzzy fields. We use variable thresholds per field (0.40-0.45). ExtractBench is designed for information extraction evaluation, which is exactly what JD parsing is.

Why we differ: Our P10 thresholds are calibrated to ACTUAL LLM performance. A fixed 0.8 would fail most company/location comparisons due to abbreviation/format differences. Our lower thresholds are more realistic but less strict.
ACKNOWLEDGED -- ExtractBench's 0.8 is a published standard. Our thresholds are looser but empirically justified.

DSPy Float Optimization

Use float scores instead of binary gates
Difference: DSPy optimizes directly on float metrics (exact_match, F1, etc.) without binary thresholds. Our compareScorecards() uses binary gate first, then float tiebreakers.

Why we differ: Our domain (CV generation) has HARD REQUIREMENTS (no fabrication, correct dates). A CV that's "95% good" but fabricates a certification is UNUSABLE. Binary gates enforce these hard requirements. Float-only optimization could produce high-average but catastrophically-flawed outputs.
JUSTIFIED -- Binary gates for hard requirements, float for fine-tuning. DSPy's pure-float approach works for benchmarks but not for production CV quality.

LLM Judge (DeepEval/Promptfoo)

Use an LLM to judge output quality
Pros: Catches nuance (tone, persuasiveness, flow). More aligned with human judgment. DeepEval, Promptfoo, and Datadog all offer LLM-judge tools.

Cons: Circular (AI judging AI). Variable (same input, different verdict between runs). Costly ($0.01-0.05 per judgment x 80 checks/iteration = $0.80-4.00/iteration). The optimization loop would need 10x the budget.

Industry consensus: LLM judges for FINAL evaluation (human proxy). Programmatic metrics for in-loop optimization.
CORRECT DECISION -- Deterministic scoring for optimization. LLM judges belong in post-loop human review.

Semantic Similarity (Embeddings)

Use embedding cosine similarity instead of word overlap
Pros: Catches synonyms ("AWS" ~ "Amazon Web Services"). More robust than string matching. BERTScore shows 59% human alignment.

Cons: Requires embedding model API calls (cost + latency). Adds dependency. "Denver, CO" and "Denver, USA" would score high -- but so would "Denver, CO" and "San Francisco, CA" (both US cities). Embeddings are too fuzzy for field-level extraction evaluation.

Status: BERTScore is deferred as Gate 2 tiebreaker. For field-level matching, word overlap + Levenshtein is sufficient and zero-cost.
DEFERRED -- BERTScore for tiebreaking is correct. For field-level matching, string methods are sufficient.
